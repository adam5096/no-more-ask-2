# 組件與邏輯收斂審查 (Unification Review)

基於「代碼重用」與「UX 一致性」原則，對 UI 組件與業務邏輯進行審查。

---

## 檢查項目

### 1. UI 原子化 (UI Atomization)

**必須檢查**：
- [ ] 是否有 2 個以上組件存在相似的 HTML 結構與 Tailwind Classes？
- [ ] 基礎元素（Input, Button, Card）是否已封裝為「原子組件」？
- [ ] 原子組件是否正確處理了全局交互模式（如 Hover 圓角、Active 縮放）？

**正確範例**：
- 抽取 `AuthInput.vue` 處理所有登入/註冊/個人資料的輸入框。

---

### 2. 業務邏輯對等性 (UX Consistency)

**必須檢查**：
- [ ] 同一業務規則（如密碼複雜度）在不同流程中是否有相同的交互反饋？
- [ ] 是否存在「註冊有提示，登入卻沒提示」的認知斷層？
- [ ] 核心驗證邏輯是否已抽離為 `Composable` 供多處呼叫？

**正確範例**：
- 在 `LoginForm` 中引入註冊階段的密碼規則標籤作為實時提示。

---

### 3. 單一真理來源 (SSOT)

**必須檢查**：
- [ ] 類型定義（Interface/DTO）是否在多個組件中重複宣告？
- [ ] 錯誤信息與驗證 Regular Expression 是否散落在各組件？

**正確範例**：
- 將 `FieldErrors` 定義在 `types/api-error.ts`。
- 將密碼檢查的正則表達式封装在 `usePasswordStrength.ts`。

---

## 審查時的判斷

| 情況 | 判定 |
|------|------|
| 散落的重複樣式 | ⚠️ 建議改進 |
| 跨流程 UX 不一致 | ❌ 需要修改 |
| 跨組件重複宣告 Data DTO | ❌ 需要修改 |
| 邏輯寫死在組件內且無法共享 | ⚠️ 建議拆分 |
