---
description: 如何定義「一對一」與「多對多」的開發邊界？
---

在 OOUX 的開發流程中，定義「一對一」或「多對多」的邊界並非單純的技術選擇，而是**業務邏輯的真相**。基數關係會徹底改變物件的定義與系統的複雜度。

以下是定義開發邊界的三個關鍵步驟：

### 1. 進行「雙向關係」測試 (Double-Sided Mapping)
不要只從一個方向思考。你必須從物件 A 看向物件 B，再從 B 看向 A。
*   **一對一 (1:1/0:1)**：如果一個「烘焙品」只能由「一位」員工研發，且該員工只負責這個產品，這就是 1:1。這在前端代表一個簡單的資料欄位或單一連結。
*   **多對多 (M:M)**：如果「員工」可以支援「多家」分店，且「分店」擁有多名「員工」。這在前端則需要處理數組（Arrays）、標籤雲或卡片列表。

### 2. 使用「具體實例」驗證邊界 (Instance Testing)
利用你採集到的名詞與實例來測試邊界是否會「破版」。
*   **練習問法**：「員工 Sophia 可以同時出現在 Midtown 和 Downtown 兩家店的排班表上嗎？」
*   如果答案是「可以」，你的 UI 必須從「單一店名」轉向「店名列表」；你的施工規範（API）則需從 `store_id` 改為 `store_ids` 陣列。

### 3. 將基數轉化為 UI 導覽藍圖
基數關係決定了使用者如何穿梭於系統中：
*   **一對一/零對一**：通常表現為「詳細資訊」中的一個跳轉連結。
*   **多對多**：通常需要一個「容器（Container）」，例如在商店頁面顯示一個員工的「卡片網格（Grid）」。
*   **預防「形狀變化」 (Shape-shifting)**：在施工規範中預先定義好這些關係，能確保你設計的組件（如員工卡片）在不同物件頁面中都能共用，且資料結構一致。

**定義這些邊界能幫助你在進入 High-fidelity 設計前取得團隊核准，避免後期的重工。**
