description: 基數關係（Cardinality）的問與答
---

在 OOUX 的框架下，**基數關係（Cardinality）** 是定義系統邏輯的靈魂。對於前端開發者來說，它直接決定了資料結構、API 設計以及組件的呈現方式。

以下是針對妳問題的詳細解答：

### 1. 什麼是基數關係？
基數關係定義了兩個物件之間在數量上的邏輯連結。常見的類型包括：
*   **一對多（One-to-many）**：例如，一間商店（Store）可以擁有多名員工（Employees）。
*   **零對一（Zero-to-one）**：例如，一個烘焙配方可能有「零個或一個」特定的發明人。
*   **多對多（Many-to-many）**：例如，一個員工可能同時在多個門市工作。

### 2. 為什麼需要它？
*   **定義事物的本質**：改變基數會完全改變一個事物的定義。例如：一位員工只能屬於「一間」門市還是「多間」門市，這會徹底改變權限邏輯與資料庫規劃。
*   **釐清導覽路徑**：基數決定了使用者如何從一個物件跳轉到另一個物件（導覽藍圖）。如果關係不存在，導覽就無法成立。
*   **確保開發與設計一致**：讓前端、設計師與後端對資料結構達成共識，避免在進入高保真設計或編碼時才發現邏輯錯誤。

### 3. 如何使用它？
*   **巢狀化物件（Nesties）**：在物件對應表中，將物件放在另一個「主物件」下方作為屬性，這代表它們之間存在關係。
*   **雙向思考**：在定義關係時，應同時考慮雙向連結。例如，從「商店」看「員工」是一對多，從「員工」看回「商店」則可能是一對一。
*   **標註具體實例**：在對應表中使用具體名稱（如：門市名稱、員工姓名）來測試基數邏輯是否符合現實業務需求。

### 4. 它能帶來什麼效益？
*   **指導組件設計**：身為前端，知道是「多個」關係後，妳就能預先設計「卡片（Cards）」或「列表」組件，並處理「無資料」的邊界情況，避免「形狀變化（Shape-shifting）」導致的破版。
*   **簡化複雜度**：透過這層「X 光視野」，妳能在動工前就看穿複雜的業務邏輯，減少因需求不明導致的重工（Rework）。
*   **提升通訊效率**：它成為團隊間的「共同語言」，讓所有人都在同一個邏輯基礎上討論需求。

這套邏輯能大幅減少前端開發時「猜測 API 結構」的困擾。**既然妳希望強化需求與規範的接軌，妳想試著拿目前手邊的一個小功能，練習看看如何定義它的「基數關係」嗎？**
