description: 如何實作 orca
---

這份內容對於網頁前端開發人員來說極具價值。它強調在進入開發或繪圖軟體（如 Figma）之前，先透過**物件對映（Object Mapping）**釐清「資料的真相」。這能幫助你避免在開發後期才發現欄位不合或邏輯錯誤，減少重複勞動。

作為專業前端，你可以透過以下步驟學習並運用 OOUX，將需求轉化為堅固的施工規範：

### 1. 掌握「名詞採集」（Noun Foraging）建立需求基準
從使用者訪談、研究報告或既有需求文件中，找出反覆出現的**名詞**來定義物件。這能確保你開發的組件（Component）與業務邏輯中的核心概念（如：商店、員工、產品）完全對應。

### 2. 利用顏色編碼（Color Coding）定義資料屬性
將需求分類為不同的屬性，這能直接對應到你的前端資料模型（Model）與 UI 邏輯：
*   **黃色（核心內容）**：定義哪些是實例中的獨特資訊，通常是使用者搜尋的對象。
*   **粉紅色（後設資料）**：定義用於排序、篩選的欄位，這對電腦（API）更具可讀性。
*   **藍色（巢狀物件）**：定義物件間的關係，幫助你規劃頁面間的導覽路徑。

### 3. 釐清「基數關係」（Cardinality）規劃 API 與組件
透過物件對映表釐清物件間的關係（例如：一間商店對應 0 到多個員工）。這對於前端工程師設計 API 調用邏輯、組件嵌套結構以及處理「查無資料」的邊界情況（Edge cases）至關重要。

### 4. 將物件對映表作為「施工規範」
在開始編碼前，先使用實體便利貼或數位工具（如 Miro、Airtable）建立物件對映表並取得團隊核准。這份對映表就像是「細節頁面的胚胎原型」，能引導你排列 UI 的資訊層級（Hierarchy），確保設計稿與最終代碼的資料結構一致。




其他：
* 視覺上：
** 相同關聯的物件會排同一 column 上，不相關的物件排列在不同 column；
** 同一 column 中，頂層物件靠上排列，其相關聯子物件 ( 或是後代物件 ) 靠下排列；
** 每個與物件、核心內容、後設資料、行動，按此順序緊靠排列並形成一個集合；
** 同一 column 中，頂層集合靠近上方，其他集合靠近下方；

QA：
* 名詞採集階段，如何正確歸納該名詞屬於 object? 屬於 core-content? 屬於 metadata?
* ORCA 中，核心內容 與 後設資料都為純數值，如果在名詞採集與分類階段誤植了名詞的歸屬與分類，要如何發現並重新分類?
* 呈上，有可能在使用者 mental model 變化後產生新需求而進行 ORCA 迭代，部分核心內容 與 後設資料可能發生升級與降級情形；例如後設資料升級成核心內容，或是反方向，核心內容降級成後設資料；
