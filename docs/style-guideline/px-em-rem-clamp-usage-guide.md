這是一份針對 **2025-2026 年現代網頁開發**所撰寫的 CSS 單位選型指南。

這份指南的核心哲學是：**「讓結構保持彈性，讓細節保持精準，讓感受保持流暢。」**

---

## 📐 2025-2026 CSS 單位選型戰術指南

### 1. 單位定義與當代定位

| 單位 | 性質 | 2026 年的定位 |
| --- | --- | --- |
| **`px`** | 絕對單位 | **物理精準層**：用於不隨縮放改變的固定裝飾。 |
| **`rem`** | 根相對單位 | **使用者尊重層**：用於確保 A11y (無障礙) 的全域基準。 |
| **`em`** | 父相對單位 | **元件局部層**：用於處理元件內部的「比例共生」關係。 |
| **`clamp()`** | 函數式邏輯 | **流體適配層**：解決 Discrete (跳變) 痛點的核心工具。 |

---

### 2. 具體場景選型建議

#### **A. 文字排版 (Typography)**

* **首選：`rem` 或 `clamp()**`
* **原因**：瀏覽器預設字體大小（通常是 16px）是使用者唯一能直接控制的視覺入口。使用 `rem` 能確保當長輩調大系統字體時，你的網頁不會變成一團亂碼。
* **2026 實踐**：標題類使用 `clamp(1.5rem, 4vw + 1rem, 3rem)`，內文類使用固定的 `1rem` 或 `1.125rem`。



#### **B. 佈局間距 (Margin, Padding, Gap)**

* **首選：`rem**`
* **原因**：間距應該隨文字大小同步放大。如果文字變大但間距不變，視覺上會顯得過於擁擠。
* **例外**：元件內部的微小間距（如 Icon 與文字間的距離）可使用 `em`，讓 Icon 隨文字縮放。



#### **C. 邊框與陰影 (Border, Box-shadow)**

* **首選：`px**`
* **原因**：`1px` 的邊框在任何裝置上都應該是細緻的一條線。如果用 `rem` 定義邊框，在大字體模式下邊框會變得粗笨，這通常不是設計師想要的。



#### **D. 元件封裝 (Component Scoping)**

* **首選：`em**`
* **原因**：當你開發一個 UI Kit（如按鈕、標籤）時，使用 `em` 定義 `padding`。
* **效果**：只要使用者在外部定義 `.btn { font-size: 2rem; }`，按鈕的內距就會自動等比擴張，無需手動調整多個 class。



---

### 3. 2026 年的「流體設計」公式 (Fluid Design)

進入 2026 年，我們應該盡量減少在 Media Queries 中手動改寫數值。

#### **推薦寫法：使用 `clamp()` 處理連續變化**

```css
/* 讓 Container 的 Padding 在手機 1rem 與桌機 4rem 之間自動流動 */
.container {
  padding-inline: clamp(1rem, 5vw, 4rem);
}

/* 讓標題在不同裝置間優雅縮放，無需 md:text-5xl */
h1 {
  font-size: clamp(2rem, 8vw, 5rem);
}

```

---

### 4. 決策流程圖 (Decision Tree)

當你在寫 CSS 屬性時，依照此路徑思考：

1. **這個屬性需要隨「系統字體大小」連動嗎？**
* **No** (例如邊框、細線) → **用 `px**`。
* **Yes** → 進入下一題。


2. **它是屬於「全域一致」還是「元件內部比例」？**
* **全域一致** (例如頁面 Padding、標準字體) → **用 `rem**`。
* **內部比例** (例如按鈕內距、Icon 大小) → **用 `em**`。


3. **它在 Mobile 到 Desktop 之間需要「連續縮放」嗎？**
* **Yes** → **用 `clamp()**`（推薦將 `rem` 作為 clamp 的參數之一）。



---

### 5. 附錄：關於 `62.5%` 技巧的 2026 觀點

> **背景**：過去開發者喜歡設定 `html { font-size: 62.5% }` 讓 `1rem = 10px` 方便計算。

* **2026 建議：不再推薦。**
* **理由**：
1. 現代開發工具（如 Tailwind, VS Code Extensions）都能自動換算 px 與 rem。
2. 這會破壞第三方元件庫（如 Vuetify, Headless UI）的樣式計算，因為它們預期根字體是 16px。
3. 直接學習 `1rem = 16px`、`0.75rem = 12px` 的倍數感，對長期維護更有利。
